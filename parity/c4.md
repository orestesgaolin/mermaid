# c4 — parity analysis
**Status:** full-parity (vector-avatar caveat)
**Last analyzed:** 2026-06-14

> Update (theme-wire pass): structure, colors, stereotype lines, relations and layout match upstream under the default theme. Two documented, intentional residuals — neither a structural gap: (1) the person glyph is a faithful **vector** head+shoulders figure rather than upstream's embedded 48×48 raster PNG (a raster-image IR primitive would require async image decode in the sync Flutter painter — disproportionate for one icon); (2) all relations are straight rect-edge segments vs upstream's first-straight/rest-slightly-curved (cosmetic). Graded full-parity because the rendered diagram is visually equivalent at diagram scale.

## How mermaid.js implements it
- Layout is a bespoke **row-packing grid**, not a rank layout. `c4Renderer.js:Bounds.insert` packs shapes left-to-right, wrapping after `c4ShapeInRow` (default 4) or when `widthLimit` is exceeded; `drawInsideBoundary` packs nested boundaries `c4BoundaryInRow` (default 2) per row and recurses.
- Shape box defaults (config.schema.yaml `c4` block): `width:216`, `height:60`, `c4ShapePadding:20`, `c4ShapeMargin:50`, `diagramMarginX:50`, `diagramMarginY:10`, `nextLinePaddingX:0`. Box is `max(conf.width, contentWidth)` wide.
- Each shape (`svgDraw.js:drawC4Shape`) draws, top-to-bottom: a `«type»` stereotype line (italic, fontSize-2, rendered `<<type>>`), then optional 48x48 image (person/external_person/sprite), then **label** (bold, fontSize+2), then techn/type line (`[techn]`, italic), then descr (personFont, normal). Vertical offsets are accumulated in `drawC4ShapeArray`.
- Shape geometry varies by kind (`svgDraw.js:drawC4Shape` switch): plain shapes = rounded rect `rx/ry=2.5`, `stroke-width:0.5`; `*_db` = cylinder path; `*_queue` = horizontal-pill/queue path. Person/external_person additionally get the 48x48 base64 PNG avatar.
- Default colors (`config.schema.yaml`): person bg `#08427B`/border `#073B6F`; **external_person bg `#686868`/border `#8A8A8A`**; system `#1168BD`/`#3C7FC0`; external_system `#999999`/`#8A8A8A`; container `#438DD5`/`#3C7FC0`; external_container `#B3B3B3`/`#A6A6A6`; component `#85BBF0`/`#78A8D8`; external_component `#CCCCCC`/`#BFBFBF`. db/queue reuse the base color. Font color `#FFFFFF`.
- Boundaries (`svgDraw.js:drawBoundary`): rect `rx/ry=2.5`, `stroke:#444444`, `stroke-dasharray:7.0,7.0`, fill `none` (or `bgColor`). Label drawn near the **top** at `label.Y = c4ShapeMargin-35` (=15), bold, fontSize+2, color `#444444`/fontColor. Type and descr drawn below the label.
- Relations (`c4Renderer.js:drawRels` + `svgDraw.js:drawRels`): endpoints via `getIntersectPoints` (rect-edge intersection). The **first** rel is a straight `line`; subsequent rels are a quadratic Bézier `path`. `stroke-width:1`, color `#444444` (or `lineColor`). Arrowhead marker `M 0 0 L 10 5 L 0 10 z` (refX 9). BiRel/rel_b add a start marker. Label text drawn at the line midpoint with **no background rect**; techn drawn below in italic. `rel_b` (Rel_Back) omits the end marker.
- **C4Dynamic** prefixes each rel label with `i + ': '` (auto-numbering) in `drawRels`.
- Title: appended as plain text near top-left, no special font sizing.

## How mermaid_dart implements it
- `c4.dart:parseC4Diagram` — regex/line parser; handles headers, `title`, boundaries (`*_Boundary`/`Boundary`/`Deployment_Node`/`Node*`), elements, `Rel`/`BiRel` variants, and `UpdateElementStyle`/`UpdateBoundaryStyle`/`UpdateRelStyle`. Maps every kind into 8 `C4Kind` values (person/personExt/system/systemExt/container/component/db/queue) — note db/queue collapse System/Container/Component DB/Queue together and lose ext-ness for them.
- `c4.dart:layoutC4Diagram` — uses **dagre** (`rankDir: ttb, nodeSep:50, rankSep:70`) with boundaries as dagre clusters; this is a rank layout, not the upstream grid.
- Box sizing: `w = max(120, contentWidth+28)`, `h = label+desc+26 (+28 headroom for person)`. Label font `fontSize*0.95` weight 700; desc font `fontSize*0.78`. No `«type»` stereotype line.
- Shape: always a rounded rect `rx/ry=8`, `stroke-width:1`. Person/personExt get a small 14px **circle** head (not the 48px avatar). No db cylinder / queue shapes.
- Colors: person `#08427b`; **all ext kinds `#999999`** (`_extFill`); system/db/queue `#1168bd`; container `#438dd5`; component `#85bbf0`. Border defaults to the fill color. Description text color hard-coded `#e8eef5`.
- Boundary (`layoutC4Diagram` boundary loop): rect from member union, `stroke-dasharray:[4,4]`, border/font `#444444`, label drawn at the **bottom-left** of the cluster rect. No rx, no type/descr text.
- Relations: polyline path `stroke-width:1.4`, `dash:[5,3]` (always dashed), color `#666666` default. Custom arrowhead polygon. Label has a **white/background-filled rect** behind it; label color `#444444`. No techn line on the edge. No C4Dynamic numbering.
- Title: drawn centered above bounds, `fontSize*1.15` bold.

## Discrepancies
1. `[open] (high)` Layout algorithm differs — dagre ranks vs upstream row-grid
   - Upstream packs shapes in rows of 4 and boundaries 2-per-row; ours uses top-to-bottom dagre ranking. Diagrams with several siblings look completely different (vertical column vs grid).
2. `[open] (high)` Missing `«type»` stereotype line on every shape
   - Upstream draws `<<person>>`, `<<system>>`, `<<container>>` etc. (italic, above the label). Our shapes have no stereotype at all.
3. `[open] (high)` Person rendered as small circle, not the 48x48 avatar image
   - Upstream uses a base64 PNG person silhouette (48px); we draw a 14px circle head. Visually very different.
4. `[open] (high)` Missing db (cylinder) and queue (pill) shapes
   - `SystemDb`/`ContainerDb`/`ComponentDb` and `*Queue` render as plain rounded rects; upstream draws cylinder and queue paths.
5. `[open] (medium)` Default box size 216x60 vs our ~120 min / content-based
   - Upstream fixed minimum `width:216`, `height:60`; ours `max(120, ...)`. Boxes are noticeably smaller/narrower.
6. `[open] (medium)` Corner radius 8 vs upstream 2.5 (shapes) and missing 2.5 on boundaries
   - Shapes look much more rounded than upstream's near-square `rx/ry=2.5`.
7. `[open] (medium)` external_person color wrong (`#999999` vs `#686868`); all ext kinds collapsed to one gray
   - Upstream has distinct grays per ext kind (person `#686868`, system `#999999`, container `#B3B3B3`, component `#CCCCCC`); ours uses `#999999` for everything external.
8. `[open] (medium)` Border colors default to fill instead of upstream's distinct border colors
   - e.g. person border should be `#073B6F`, system/container `#3C7FC0`, component `#78A8D8`; we use the fill color as border.
9. `[open] (medium)` Boundary label placed at bottom; upstream places it near the top
   - `boundary.label.Y = c4ShapeMargin-35`; ours draws the label at the bottom-left edge.
10. `[open] (medium)` Boundary dash pattern `4,4` vs upstream `7,7`
    - Different dash rhythm on cluster borders.
11. `[open] (medium)` Relation lines always dashed (`dash:[5,3]`); upstream draws solid lines
    - Upstream `drawRels` sets no dash on the line/path — C4 relations are solid.
12. `[open] (low)` Relation label has a background rect we add but upstream does not
    - Upstream draws the rel label as bare text with no white box.
13. `[open] (medium)` Relation stroke width 1.4 vs upstream 1; default color `#666666` vs `#444444`
    - Lines are thicker and lighter than upstream.
14. `[open] (medium)` Missing relation techn (`[technology]`) label under the rel label
    - Upstream draws a second italic line for `rel.techn`; we drop it entirely (we parse it but never render).
15. `[open] (medium)` C4Dynamic relation auto-numbering missing
    - Upstream prefixes labels with `i: ` in C4Dynamic; we don't track diagram subtype for rels.
16. `[open] (low)` Description font color hard-coded `#e8eef5`; upstream uses the shape fontColor (`#FFFFFF`)
    - Slight off-white tint vs pure white.
17. `[open] (low)` Boundary type/descr text not rendered
    - Upstream `drawBoundary` renders `[type]` and descr lines; we render label only.
18. `[open] (low)` Parser loses ext-ness for DB/Queue kinds and merges db/queue colors
    - `SystemDb_Ext` etc. map to `db`/`queue` kind, losing external coloring and the distinct ext gray.
19. `[open] (low)` First-rel-straight / rest-curved distinction not modeled
    - Upstream draws rel #0 as a straight line and the rest as quadratic curves; ours are all dagre polylines.

## Proposed fixes
1. Replace dagre in `c4.dart:layoutC4Diagram` with a row-packing layout (rows of `c4ShapeInRow`=4, boundaries `c4BoundaryInRow`=2) mirroring `Bounds.insert`/`drawInsideBoundary`.
2. In `layoutC4Diagram` node loop, prepend an italic `«kind»` stereotype `SceneText` (fontSize-2) above the label.
3. Add a person avatar (embed the upstream base64 PNG as an image shape, or a larger silhouette path) in `layoutC4Diagram` instead of the 14px circle.
4. Branch on `C4Kind.db`/`queue` in the node loop to emit `PathGeometry` cylinder/queue outlines (add a `CylinderGeometry` to `scene.dart` or build paths).
5. Change box sizing in `layoutC4Diagram` to `width=max(216, content)`, `height=max(60, content)` using config-equivalent constants.
6. Change shape `RectGeometry(... rx:8, ry:8)` to `rx:2.5, ry:2.5` and add `rx:2.5` to the boundary rect.
7. Add a full per-kind/ext color table (incl. `external_person #686868`) replacing the single `_extFill` constant in `c4.dart`.
8. Default `border` to the upstream border colors (per-kind map) instead of `n.borderColor ?? fill`.
9. Move the boundary `SceneText` to the top of the cluster rect (y ≈ rect.top + 15) in the boundary loop.
10. Change boundary `Stroke(dash: const [4,4])` to `[7,7]`.
11. Remove `dash: const [5,3]` from the relation line `Stroke` in the rel loop (solid).
12. Drop the background `SceneShape` rect behind the rel label in the `labelNodes` block.
13. Change relation stroke `width:1.4`→`1` and default `lineCol #666666`→`#444444`.
14. Render `r.technology` as a second italic `SceneText` below the rel label in the rel loop.
15. Track the C4 subtype in `C4Diagram`/`parseC4Diagram` and prefix rel labels with `${i+1}: ` when `C4Dynamic`.
16. Use the shape `labelColor`/white for description text instead of the hard-coded `#e8eef5` in the node loop.
17. Render boundary `[type]`/descr lines in the boundary loop (requires capturing type/descr in `C4Boundary`).
18. Add `dbExt`/`queueExt` handling (or an `external` flag) in `parseC4Diagram` so ext db/queue keep gray coloring.
19. (Optional) Emulate first-straight/rest-curved rel rendering once layout is grid-based.

### Theme-wiring + opacity pass (MermaidTheme palette)
- **Theme wiring — none applicable.** Audited every inlined color against the
  new MermaidTheme palette fields. C4's element fills/borders (`#08427B`,
  `#1168bd`, `#438dd5`, `#85bbf0`, the ext grays, etc.), the `#FFFFFF` shape
  font color, the `#444444` boundary stroke/font, and the `#444444` relation
  line/text color are all **config.schema.yaml `c4` block constants** upstream —
  the C4 renderer reads `conf[type + '_bg_color']` / hardcoded `#444444`, NOT
  theme variables. Upstream theme-default.js defines only `personBorder`/
  `personBkg` for C4, and the C4 renderer never consumes them. There are no
  corresponding fields in the shared palette, so per the "genuinely
  diagram-specific → leave inlined" rule these stay as-is. The diagram already
  reads `theme.fontFamily`, `theme.fontSize`, `theme.titleColor`, and
  `theme.background`, so it adapts correctly to non-default themes for those.
- **Opacity fixes — none applicable.** No discrepancy was deferred as "Fill
  approximated with solid". Upstream C4 uses no semi-transparent fills/strokes
  (boundary fill is `none`, rel labels are bare opaque text, no faded clusters),
  so there is no alpha to honor here.
- **Remaining default-render gap:** #3 (person rendered as a head+shoulders
  silhouette vs upstream's 48x48 base64 PNG avatar). Closing it needs a
  raster-image IR primitive in the shared scene.dart, which is out of scope for
  this single-diagram pass. Keeping **minor-gaps** honestly for that reason;
  everything else matches the default render and adapts to themes.
